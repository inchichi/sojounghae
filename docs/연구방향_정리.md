# 선행연구 대비 연구방향 정리

## 1. 우리가 먼저 분명히 해야 할 점

이 연구의 목표는 단순히 `고객 이탈을 맞히는 모델`을 만드는 데 있지 않다.
우리가 실제로 추구해야 하는 것은 다음이다.

- 어떤 고객이 이탈하는지 예측한다.
- 왜 이탈하는지 설명한다.
- 무엇을 바꾸면 이탈을 줄일 수 있는지 제안한다.

즉, 이 연구는 `예측 성능`만 보는 작업이 아니라 `유지 전략`까지 연결되는 연구여야 한다.

---

## 2. 선행연구들의 공통 한계

문헌 검토를 기준으로 보면, 기존 연구들은 대체로 다음 한계를 가진다.

### 2.1 정확도 중심 평가에 치우침

많은 연구가 `AUC`, `Accuracy`, `F1` 같은 통계적 성능만 강조한다.
그러나 이탈 예측은 실제 비즈니스 의사결정과 연결되어야 하므로,
모델이 얼마나 맞히는지뿐 아니라 `왜 그런지`, `어떤 조치를 취할지`가 중요하다.

### 2.2 단일 시점 데이터에 의존

대부분의 연구는 한 시점의 정적 데이터만 사용한다.
이 방식은 시간에 따른 고객 행동 변화나 `concept drift`를 충분히 반영하지 못한다.

### 2.3 데이터셋 규모와 범위가 제한적

IBM Telco처럼 널리 쓰이는 데이터셋은 있지만, 상대적으로 규모가 작고
실제 통신 환경의 복잡성을 충분히 담지 못할 수 있다.

### 2.4 해석 가능성이 부족함

정확도가 높더라도 블랙박스 모델로 끝나면 실무 활용도가 낮다.
실무에서는 `어떤 변수 때문에 위험한지`를 바로 설명할 수 있어야 한다.

### 2.5 일반적인 피처만 사용

기존 연구는 표준적인 변수들만 쓰는 경우가 많고,
도메인 지식을 반영한 상호작용 변수나 비용 충격 변수는 충분히 다루지 않는다.

### 2.6 가설 검증이 약함

문제 자체를 설명하는 연구질문은 많지만,
명확한 가설을 세워 비교·검증하는 연구는 상대적으로 적다.

---

## 3. 우리가 선행연구에 비해 추구해야 할 것

### 3.1 비즈니스 정렬

우리는 이탈 예측을 단순 분류 문제가 아니라
`고객 유지 전략 수립 문제`로 봐야 한다.

따라서 모델 성능이 좋다는 이유만으로 끝내면 안 되고,
실제로 운영에서 사용할 수 있는 해석과 액션까지 도출해야 한다.

### 3.2 설명 가능한 머신러닝

우리 연구는 반드시 설명 가능해야 한다.

중요한 이유:

- 서비스 품질이 문제인지
- 비용 충격이 문제인지
- 충성도 저하가 문제인지

이런 질문에 답해야 실무적으로 의미가 생긴다.

### 3.3 도메인 기반 피처 엔지니어링

우리가 가진 강점은 Cell2Cell 데이터셋의 풍부한 변수다.
여기서 단순 원본 변수만 쓰지 말고, 도메인 지식을 반영한 변수를 만들어야 한다.

예시:

- `PositiveRevenueShock`
- `OverageMinutes x PositiveRevenueShock`
- `RoamingCalls x PositiveRevenueShock`
- `OverageMinutes x RoamingCalls`
- `Revenue per Month`
- `Call Drop Rate`

### 3.4 베이스라인부터 앙상블까지 체계적으로 비교

한 모델만 돌리는 방식이 아니라,

- `Logistic Regression`으로 기준선을 만들고
- `Random Forest`로 안정성을 보고
- `XGBoost`로 성능 상한을 확인해야 한다

이렇게 비교해야 연구의 설득력이 생긴다.

### 3.5 성능뿐 아니라 해석과 안정성까지 확인

단일 점수 최고값보다 중요한 것은 다음이다.

- 여러 fold에서 일관되게 좋은가
- 엔지니어링된 피처가 실제로 도움이 되는가
- 해석 결과가 도메인 상식과 맞는가

---

## 4. 우리가 세워야 할 연구 방향

### 방향 1. 이탈 예측과 유지 전략을 연결한다

모델 출력은 단순히 `이탈/비이탈`이 아니라,
`누구를 우선적으로 붙잡아야 하는가`로 이어져야 한다.

즉, 연구 결과는 다음 형태로 정리되어야 한다.

- 위험 고객군 식별
- 위험 요인 순위화
- 우선 개입 변수 제안

### 방향 2. 비용 충격과 서비스 품질을 핵심 축으로 본다

우리 데이터에서 특히 중요한 축은 다음이다.

- 사용량 증가와 초과 사용에 따른 비용 충격
- 로밍 및 고비용 사용 패턴
- 통화 품질 저하
- 고객센터 문의와 불만족 신호

이 축을 중심으로 이탈 메커니즘을 설명해야 한다.

### 방향 3. 해석 가능한 성능 개선을 목표로 한다

목표는 무조건 최고 성능이 아니다.

정확히는:

- 기본 모델보다 나아지고
- 해석 가능한 수준을 유지하면서
- 실제로 설명 가능한 개선을 만드는 것

이게 더 중요한 연구 목표다.

### 방향 4. 가설 기반 연구로 간다

우리 연구는 다음처럼 명시적인 가설 구조를 가져가야 한다.

- 앙상블이 베이스라인보다 낫다
- 엔지니어링 피처가 성능을 높인다
- 서비스 품질 변수의 중요도가 높다
- 비용 충격 변수는 이탈과 유의한 관계를 가진다

가설이 있어야 결과 해석이 흔들리지 않는다.

---

## 5. 머신러닝 연구는 어떻게 진행해야 하는가

### 5.1 데이터 처리 원칙

반드시 지켜야 할 원칙은 다음과 같다.

- 전처리는 train 데이터 기준으로만 fit한다
- test 정보가 들어가지 않도록 한다
- 결측치와 이상치는 별도로 관리한다
- 범주형 변수는 일관된 방식으로 인코딩한다

### 5.2 전처리 순서

권장 순서는 다음과 같다.

1. 타깃과 입력 변수 분리
2. train/test 분할
3. train 기준 결측치 대체
4. 이상치 클리핑
5. 필요한 변수에 `log1p` 변환
6. `RobustScaler` 적용
7. 범주형 인코딩

### 5.3 피처 엔지니어링

원본 변수만 쓰지 말고 도메인 의미가 분명한 피처를 만들어야 한다.

우선순위가 높은 피처:

- 초과 사용 관련 변수
- 로밍 관련 변수
- 매출 변화 관련 변수
- 통화 품질 관련 변수
- 서비스 지속 기간 관련 변수

핵심은 `상호작용`이다.
단일 변수보다 조합이 더 강한 신호를 줄 수 있다.

### 5.4 모델링 전략

모델은 역할을 분리해서 써야 한다.

- `Logistic Regression`
  - 해석 기준선
  - 방향성과 계수 확인용

- `Random Forest`
  - 비선형 관계 확인
  - 변수 중요도 확인용

- `XGBoost`
  - 최종 성능 비교용
  - 복합 패턴 탐지용

### 5.5 평가 전략

단순 정확도만 보면 안 된다.

최소한 다음은 확인해야 한다.

- `ROC-AUC`
- `Accuracy`
- `Precision`
- `Recall`
- `F1-score`

추가로 봐야 할 것:

- fold별 성능 분산
- 임계값 변화에 따른 성능 변화
- 엔지니어링 피처 유무에 따른 차이

### 5.6 해석 전략

최종 결과는 반드시 해석해야 한다.

해석 포인트:

- 로지스틱 회귀 계수
- Random Forest / XGBoost 중요도
- SHAP 기반 기여도

해석 결과는 다음 질문에 답해야 한다.

- 어떤 고객이 위험한가
- 왜 위험한가
- 어떤 변수에 개입해야 하는가

---

## 6. 우리가 최종적으로 얻어야 할 산출물

이 연구의 최종 산출물은 단순한 모델 하나가 아니라 다음이어야 한다.

- 이탈 예측 모델
- 핵심 이탈 요인 순위
- 비용 충격과 서비스 품질의 관계 설명
- 엔지니어링 피처의 효과 검증
- 고객 유지 전략 제안

즉, `예측 모델 + 설명 + 실행 전략`이 함께 나와야 한다.

---

## 7. 한 문장으로 정리하면

우리는 선행연구처럼 `이탈을 맞히는 것`에서 끝내지 말고,
`왜 이탈하는지 설명하고, 무엇을 바꾸면 유지되는지 제안하는 설명 가능한 머신러닝 연구`를 해야 한다.
